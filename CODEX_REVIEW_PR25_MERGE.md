**Findings**
- **blocker, before enabling external consent**: consent-enabled web leads run two routing passes and can use stale `routeVia`. `createProspect` first calls `resolveLeadRouting` at [prospectService.js:201](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:201), which can advance the campaign cursor in `resolveLeadRouting` at [systemAgent.js:153](/private/tmp/mktr-pr25/backend/src/services/systemAgent.js:153). If `allowExternal` is true, it then calls `resolveLeadAssignment` at [prospectService.js:397](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:397), which advances the same cursor again at [systemAgent.js:271](/private/tmp/mktr-pr25/backend/src/services/systemAgent.js:271). For internal results, only `assignedAgentId` is overwritten at [prospectService.js:407](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:407); `routeVia` is not. The quota gate then uses the stale `routeVia` at [prospectService.js:429](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:429). With a mixed two-candidate ring, this can systematically skip every other candidate; with QR/group routing it can also treat a System fallback as `qr` and try to charge/deliver instead of quarantining as fallback.

- **should-fix**: `chargeLeadCredit` is safe against negative balances/double-spend, but not race-safe for availability. The CTE uses `FOR UPDATE ... SKIP LOCKED` at [leadCredits.js:161](/private/tmp/mktr-pr25/backend/src/services/leadCredits.js:161). If one package assignment row has multiple remaining leads, concurrent charges skip the locked row and can fall through to `owed_leads_count` or return false at [leadCredits.js:186](/private/tmp/mktr-pr25/backend/src/services/leadCredits.js:186), quarantining a lead even though credits remain after the first transaction commits.

- **should-fix, before external leads are live**: external leads are not protected in manual assignment paths. Normal assign sets only `assignedAgentId` at [prospectService.js:871](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:871), without clearing or rejecting `externalAgentId`, so the DB CHECK will reject assigning an external lead. Unassign suppresses only quarantined leads at [prospectService.js:796](/private/tmp/mktr-pr25/backend/src/services/prospectService.js:796), not external leads, so an external-only lead can emit a Lyfe `lead.unassigned` event even though Lyfe never received `lead.created`.

- **nit**: migration 035 uses targeted duplicate-error handling, but migration 034 still swallows any `addColumn` failure at [034-add-campaign-tiktok-pixel-id.js:18](/private/tmp/mktr-pr25/backend/src/database/migrations/034-add-campaign-tiktok-pixel-id.js:18). Since `runMigrations` records success after `up()` returns at [runMigrations.js:59](/private/tmp/mktr-pr25/backend/src/database/runMigrations.js:59), a non-duplicate DDL failure could be marked applied without the column.

**Sound Areas**
- Current production traffic cannot hit the external branch through the public route: `prospectCreate` does not whitelist `consentMetadata` at [validation.js:130](/private/tmp/mktr-pr25/backend/src/middleware/validation.js:130), and `externalConsent.js` documents that no current writer populates it.
- In the actual external branch, create-time mutual exclusion is correct: external selection nulls `assignedAgentId`, writes `externalAgentId`, skips `decideAssignment`, charges via `deductExternalLeadBalance` inside the transaction, and suppresses Lyfe `lead.created`.
- `deductExternalLeadBalance` is race-safe for the external wallet via a conditional `UPDATE ... leadBalance >= amount` at [leadCredits.js:111](/private/tmp/mktr-pr25/backend/src/services/leadCredits.js:111).
- Retell and Meta lead-ad callers do not bypass hard quota; both use `resolveLeadRouting` plus `decideAssignment`/`chargeLeadCredit`.
- Frontend Meta/TikTok tracking composition looks correct: ViewContent fire-once refs are honored, `ensureFbp()` is gated and runs before Meta ViewContent, Lead fires only after successful submit, and quiz reveal uses the shared CompleteRegistration dedup id.

**Verdict**
Safe for current production only because external consent is inert. I would not ship this as an active MKTR Leads external-buyer marketplace or enable consent capture until the double-routing/stale-`routeVia` blocker is fixed.
